## 一、核心洞察

**现有所有信息产品都在回避一个根本问题：信息天然不平等，但没有任何产品帮用户量化和追踪这种不平等。**

在 Obsidian/Notion/任何笔记工具里，一篇微信公众号文章和一次专家一手访谈长得一模一样——都是一个 markdown 文件。但它们对用户判断的影响应该完全不同。

Cortex 不是笔记工具，不是搜索引擎，不是知识管理系统。

**Cortex 是一个认知基础设施——帮助用户在信息洪流中保持清醒的判断力。**

## 二、信息框架

### 2.1 三个固有维度

每条信息有三个在输入时可确定的固有属性：

| 维度 | 回答什么 | 决定什么 | 示例 |
|---|---|---|---|
| **来源** | 从哪来的？ | 可信度权重 | 一手经历 > 专家对话 > 朋友转发 > 刷到的文章 |
| **性质** | 是什么东西？ | 系统如何处理 | 观点/判断、事实/数据、方法/工具、问题、直觉、综合结论 |
| **时效** | 什么时候失效？ | 信息生命周期 | 永久事实、持续趋势、时效性情报、可验证预测 |

### 2.2 信息性质的六种类型

| 类型 | 例子 | 系统处理方式 |
|---|---|---|
| **观点/判断** | "去银化会加速" | 关联 thesis，追踪支持/反对 |
| **事实/数据** | "Q1 出货量 3.2GW" | 记录，可验证，可被引用 |
| **方法/工具** | "用这个框架分析 TAM" | 索引备查，不参与信念追踪 |
| **问题** | "为什么突然换 CTO？" | 转化为关注方向，新信息进来时自动匹配 |
| **直觉** | "总觉得不对" | 保留，等待印证或消散 |
| **综合结论** | "我现在认为..." | 认知里程碑，用来跟未来的自己对比 |

### 2.3 动态价值计算

信息的真正价值不是固有的，而是相对于用户当前知识状态动态计算的：

```
信息 = 来源 × 性质 × 时效   （固有，输入时确定）
价值 = f(信息, 用户知识状态)  （动态，持续计算）
```

同一条信息可能是：
- **新信号**：拓展认知边界
- **重复信号**：不增加价值
- **矛盾信号**：挑战现有判断
- **解答**：回应之前的开放问题
- **桥接**：连接之前不相关的两个认知

这个计算不是一次性的。每次知识状态变化，所有历史信息的"相对位置"都可能改变。

### 2.4 关键原则

**判断权永远在用户手里。系统不替用户认知。**

系统不告诉你"你的置信度是 61%"。系统把你自己都看不清的证据地形图铺在你面前，让你自己判断。

用户在输入信息时的自然语言反应（"有道理"、"扯淡"、"留着看看"）是最好的元数据，比任何标签系统都真实。

## 三、目标用户

### 3.1 核心场景

**在不确定性中做高价值决策的人。**

核心痛点不是"找不到信息"，是"信息太多，判断在退化"。

### 3.2 优先级排序

| 用户群 | 痛感 | 付费意愿 | 优先级 |
|---|---|---|---|
| VC / 买方研究员 | 极强（错过信号 = 真金白银） | 高 | **起点** |
| 卖方分析师 / 行业研究员 | 强（碎片 → 结构化的加速需求） | 中高 | 第二圈 |
| 创业者 / 产品负责人 | 中强（市场假设需要持续验证） | 中 | 扩展 |
| 调查记者 / 深度内容创作者 | 强但小众 | 低 | 长期 |
| 个人知识管理爱好者 | 弱（享受整理过程） | 低 | 不优先 |

### 3.3 不只是 VC 工具

如果 Cortex 只是"帮 VC 管笔记"，天花板很低。核心价值是**帮所有人在信息时代保持清醒**——VC 是最锋利的切入点，但概念是普适的。

## 四、产品架构

### 4.1 三层架构

```
输入层 ──→ 智能层 ──→ 输出层
(在用户      (核心引擎,     (推送 +
 已有的       用户不直接     可探索的
 场景中)      接触)          界面)
```

**Cortex 的身份是智能层本身。** 它不是一个你"打开"的 app，而是一个持续运转的认知引擎，通过各种触点跟你交互。

### 4.2 输入层：在用户已经在的地方

核心原则：**零摩擦输入，信息在产生的地方被捕获。**

| 入口 | 交互方式 | 状态 |
|---|---|---|
| **微信** | 转发文章/消息给 Cortex，加一句评论 | NOT STARTED |
| **OpenClaw / Agent** | 作为 skill 被 Agent 调用导入信息 | NOT STARTED |
| **浏览器插件** | 读完文章，侧边栏一键 clip | NOT STARTED |
| **语音** | 语音备忘，走在路上随时记录 | NOT STARTED |
| **邮件** | 转发邮件给 Cortex | NOT STARTED |
| **Obsidian** | 保存笔记时自动同步 | DONE（Phase 1） |
| **URL 导入** | cortex import-link / POST /events/ingest | DONE（Phase 2） |
| **文件导入** | cortex import-file (PDF/DOCX/TXT) | DONE（Phase 2） |

**用户在输入时可以（但不强制）附带自然语言反应：**
- "有道理" → 系统理解：认同
- "留着看看" → 系统理解：资源收录
- "这跟上次那个专家说的对上了" → 系统理解：印证，关联历史
- 什么都不说 → 系统自己判断类型，不追问

**主界面也应有输入端**：用户有时需要严肃写作——写投资备忘录、整理 thesis、系统性思考。这类输入是"综合结论"类型，权重最高。

### 4.3 智能层：核心引擎

接收到输入后的处理流程：

```
信息输入
  ↓
1. 内容解析（提取实体、观点、数据）
  ↓
2. 三维分类（来源 × 性质 × 时效）
  ↓
3. 用户反应解读（自然语言 → 态度标记）
  ↓
4. 知识图谱关联（thesis、实体、已有信息）
  ↓
5. 动态价值计算（新信号？重复？矛盾？解答？）
  ↓
6. 触发判断（是否需要通知用户？）
```

### 4.4 输出层：推送 + 探索 + 写作

**推送（主动，系统发起）：**
- 矛盾检测："新信息与你现有判断矛盾"
- 问题解答："你之前的问题可能有答案了"
- 信号聚合："某实体本周被异常频繁提到"
- thesis 健康："某 thesis 14 天没有新数据"
- 每日/每周摘要

推送渠道：微信消息、邮件、通知（跟输入层共用渠道）

**探索（用户发起或推送触发）：**
- 以 thesis 为中心的证据地图
- 实体关系图谱
- 信念演化时间线
- 证据权重可视化

交互范式是 **Zoom**：
```
Feed（鸟瞰：什么值得注意）
  ↕ 展开
Graph（中景：关系和模式）
  ↕ 点击
Document（特写：源文件 + 上下文标注）
```

**写作（用户发起）：**
- 投资备忘录辅助
- Thesis 框架整理
- 系统性思考输出
- 写作过程中自动关联相关证据

## 五、输入形式与处理

### 5.1 支持的输入形式

| 输入形式 | 例子 | 处理方式 |
|---|---|---|
| **文字** | 直接打的一句话、一段感悟、投资备忘录 | 直接作为 event 内容 |
| **链接** | 公众号文章、网页、推文 | 抓取原文 → 提取正文 → 保留原始 URL |
| **语音** | 语音备忘、会议录音 | 转写 → 文本进入知识层，原始音频保留 |
| **图片** | 截图、白板照片、名片、图表 | OCR / 图像理解 → 文本，原图保留 |
| **文件** | PDF 研报、Word 协议、Excel 数据 | 文本提取 → 进入知识层，原文件保留 |
| **视频** | 路演录像、产品 demo、访谈 | 转写 + 关键帧提取 → 文本，原视频保留 |

### 5.2 统一处理模式

形式不同，但进入 Cortex 后需要的东西是一样的：**一段可以被理解、被嵌入、被关联的文本 + 原始内容的留档。**

```
原始输入 ──→ 转化为文本 ──→ 进入知识引擎
   ↓
  保留原件（可检索的文件库）
```

### 5.3 输入时捕获用户反应

在输入的那一刻问用户的判断或态度是最佳时机——上下文最鲜活，反应最真实。

- 不强制每条都回应
- 系统根据信息性质智能决定是否询问：
  - **信号型信息**（文章、研报）→ 适合问态度：[有道理] [不认同] [存疑] [跳过]
  - **资源型信息**（教程、工具）→ 不问，安静收录
  - **模糊地带**（既是工具介绍也涉及行业趋势）→ 轻量询问：[当参考收录] [关联到 thesis]
- 用户的自然语言评论本身就是最好的反应："有道理"、"扯淡"、"留着看看"

## 六、存储架构

### 6.1 设计原则

- **Event 是原子单元**：所有信息以 event 为中心组织，不引入独立的 claim 实体
- **原件可检索**：用户需要找到原始文件（"把汉鼎的年报找出来发给律师"），原件不是纯备份
- **知识图谱是主组织**：真正的分类、关联、搜索通过知识层完成，不依赖文件夹结构
- **两条路都能找到文件**：主路通过知识层查询，备路通过人可读的文件结构

### 6.2 三层存储

```
原件层（文件系统）──── 可检索的文件库，人可读
  ↕ 双向引用
事件层（Event，原子单元）── 所有信息的中心枢纽
  ↕ 提取
图谱层（Entities + Relations + Theses）── 原子知识网络
```

#### 原件层（Blob Store）

原始文件原封不动保存，按输入类型分目录，文件名保持人可读：

```
/store/
  ├── articles/      -- 抓取的网页/文章原文
  ├── documents/     -- 研报、协议、年报、PPT
  ├── recordings/    -- 录音、语音备忘
  ├── images/        -- 截图、照片
  └── videos/        -- 视频文件
```

文件名规则：`{主题/来源}-{日期}.{ext}`，例如：
- `汉鼎科技-2025年报.pdf`
- `睿衡光伏-专家访谈-20260308.m4a`
- `HJT银耗分析-券商研报-20260315.pdf`

不按公司/thesis 建子目录——一份文件可能跟多个主题相关，文件夹层级无法表达多对多关系，知识图谱可以。

#### 事件层（Event Store）

Event 是 Cortex 的基本单元，往上连着原件，往下连着图谱，自身携带多种表征：

```sql
events:
  id, created_at, updated_at, workspace_id

  -- 原始输入
  raw_input_type       -- text/link/audio/image/file/video
  raw_input_ref        -- 指向原件的路径/URL
  content              -- 处理后的全文文本（转写/提取/原文）

  -- 多种表征（同一 event 的不同视图）
  summary              -- AI 生成的摘要（适合人快速阅读）
  embedding            -- 向量嵌入（适合语义搜索）
  key_points           -- 关键断言列表 JSON（适合跨源对比和矛盾检测）
  stance               -- 对特定话题的立场 JSON（适合方向性判断）

  -- 三个固有维度
  source_type          -- first_hand/expert/curated/published/ambient
  source_weight        -- 0.0-1.0 可信度权重
  nature_tags          -- [claim, fact, method, question, intuition, synthesis]
  temporality          -- permanent/trend/time_sensitive/prediction
  expires_at           -- 时效性信息的过期时间（可选）

  -- 用户反应
  user_annotation      -- 用户原始自然语言评论（"有道理"、"扯淡"）
  user_stance          -- agree/disagree/uncertain/skip/null
```

**Event 的多种表征说明：**

| 表征 | 用途 | 说明 |
|---|---|---|
| `content` | 完整内容 | 最完整，用于深度阅读 |
| `summary` | AI 摘要 | 适合 feed 展示、快速浏览 |
| `embedding` | 向量嵌入 | 语义搜索、相似度计算 |
| `key_points` | 关键断言列表 | 跨源对比、矛盾检测（"两篇都提到银耗数据但数字不同"） |
| `stance` | 话题立场 | 方向性判断（"这篇对去银化 bearish，那篇 bullish"） |

**key_points 示例：**
```json
[
  {"text": "HJT 银耗已降至 15mg/片", "type": "data"},
  {"text": "预计 2026 年底降至 8mg", "type": "prediction"},
  {"text": "银包铜替代率将达 40%", "type": "claim"}
]
```

**stance 示例：**
```json
{
  "去银化进展": "bearish",
  "HJT 技术路线": "neutral",
  "投资时机": "cautious"
}
```

key_points 中的每条断言可以有自己的 embedding（用于跨 event 精确匹配），但它不脱离 event 独立存在——想看上下文，回到 event 即可。同一 event 中的 key_points 天然共享上下文和论证逻辑，不会因拆分而丢失关联。

#### 图谱层（Knowledge Graph）

从 event 中提取的结构化知识网络：

```sql
entities:     实体（公司、人、技术、概念）+ embedding
relations:    关系（实体-实体、实体-event）+ 关系类型 + 权重
theses:       用户的投资/研究主题 + 关联的 events
annotations:  用户在不同时间点对特定 thesis/entity 的判断记录（信念演化轨迹）
```

### 6.3 信息流全景

```
用户输入（微信/语音/文件/链接/文字）
  ↓
┌─────────────────────────────────────┐
│ 1. 原件保存 → /store/{type}/       │
│ 2. 内容转化 → 文本                  │
│ 3. 创建 Event                       │
│    - 生成 summary, embedding        │
│    - 提取 key_points, stance        │
│    - 分类三维度（来源×性质×时效）      │
│    - 解读用户反应                    │
│ 4. 图谱更新                         │
│    - 提取/关联 entities             │
│    - 建立 relations                 │
│    - 关联 theses                    │
│ 5. 动态价值计算                      │
│    - 新信号 / 重复 / 矛盾 / 解答？   │
│ 6. 触发判断                         │
│    - 是否通知用户？                  │
└─────────────────────────────────────┘
  ↓
输出（推送 / 探索界面 / 原件检索）
```

### 6.4 Obsidian 兼容

- Obsidian 是一个**输入渠道**（vault sync，已有）
- Obsidian 不是存储层
- 可从 Cortex 导出 markdown 回 Obsidian（可选的渲染层，不是主存储）
- Cortex 知识图谱是 source of truth

## 七、AI-Native 体现

### 7.1 不是"加了 AI 的笔记工具"

传统产品：人组织信息 → 人检索信息 → 人得出结论
Cortex：信息自己组织 → 知识主动找人 → 判断持续进化

### 7.2 具体体现

| 传统产品 | Cortex |
|---|---|
| 用户手动打 tag、建链接 | 系统自动提取实体、关联、分类 |
| 用户主动搜索 | 系统主动推送相关发现 |
| 所有信息同等对待 | 信息有来源权重、性质分类、时效管理 |
| 用户自己发现矛盾 | 系统自动检测矛盾并提醒 |
| 历史信息沉入遗忘 | 旧信息在新上下文中被重新激活 |
| 笔记就是终点 | 每条输入都在更新整个知识图谱 |

### 7.3 系统有观点，但判断权在人

系统会说："这 5 篇文章来自 3 个独立信源，都在说同一件事"（这是有价值的观察）。

系统不会说："你应该相信这个"（这是越权）。

**一个微信入口 + 智能处理 + 微信推送。**

核心循环：输入 → 理解 → 关联 → 主动反馈

### 9.2 已完成 / 尚未完成

| 模块 | 状态 | 说明 |

## 十、护城河

**用户的个人知识状态是独一无二的，且随时间持续积累。**

用得越久越有价值，因为系统理解 YOUR 信念、YOUR 知识缺口、YOUR 关注模式。任何竞品可以复制功能，但无法复制 6 个月积累的个人知识状态。

## 十一、开放问题

1. **冷启动**：第一条信息进来时，系统没有已有知识可关联，价值感弱。是否通过导入 Obsidian vault 解决？
2. **信息加工层次**：原始数据 vs 分析 vs 解读 vs 叙事——是否需要作为独立维度追踪？
3. **协作场景**：投资决策常由团队做出（IC 会议、合伙人讨论），共享信念追踪如何设计？
4. **主界面写作端**：用户需要严肃写作时的交互形式是什么？编辑器？对话？结构化模板？
5. **推送频率和形式**：多频繁推送不会让用户觉得烦？什么级别的发现值得打断？
